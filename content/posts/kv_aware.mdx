previously, for my system I was hashing an entire prompt and checking for it across all my tpus. however, this is not necessarily a good idea because:
    - this doesn't take into account if certain kv pairs got evicted from a worker's memory
    - we want to look at smaller prefixes so multi-turn chats and shared system prompts actually score

the fix is to operate at block granularity — chunk the prompt the same way vllm does, hash each block, and ask redis "which workers have recently been routed to for this block?" the trick is keeping the router's notion of what's cached on each worker in sync with reality without subscribing to a firehose of events from the engine.

## naive load-balancing

`slot count` is how many requests a worker is currently processing. max capacity is how much the worker can handle before overloading and this is hardcoded for each tpu pod type in my system. without kv awareness, the score for each worker is just

```
util = (slots + 1) / max
```

and we sort by that. two non-obvious bits in this formula:

the `+ 1` baseline matters. without it, a serial workload (each request awaits its own response before the next is issued) sees `slots = 0` for every engine at sort time, every util multiplies down to 0, and the stable sort just collapses onto endpoint[0]. with the baseline, idle engines still differentiate by `1/max`.

with kv-cache awareness, a worker that already has K overlapping prefix blocks gets its util multiplied by `0.5^K`. so an engine with eight cached blocks looks 256x less loaded than the same engine with no cache hit. this has to be done in `f64`: `0.5_f32.powi(178) ≈ 2.6e-54` underflows to zero in f32 (smallest normal ≈ 1.4e-45), which on a long shared prefix silently kills the discount and you're back to picking endpoint[0].

```rust
let util = (slots as f64 + 1.0) / max as f64;
let factor = 0.5_f64.powi(overlap as i32);
let effective = util * factor;
```

one more thing: shuffle the candidate list before sorting. ties are the norm at zero load — every engine has the same util, the same overlap — and rust's stable sort preserves input order on ties. without the shuffle, every concurrent wave of requests sorts identically and pins to endpoint[0]; with it, ties break randomly across waves.

## the fundamental shift: speculative writes

the obvious design for "which worker has block X cached" is to subscribe to the engine. vllm publishes `kv_events` over zmq whenever a block is stored, removed, or cleared — spin up a subscriber on the router, decode the event stream, populate a redis index, done. i tried this. ~2800 lines of subscriber + decoder + replay + dedupe got us to 78.95% aggregate hit rate.

then i deleted all of it and the rate went up to 83.23%.

the problem with subscribing to events: vllm only emits `BlockStored` on a *miss*, when a block is freshly populated. cache *hits* — exactly the events the router cares about most — produce no event because nothing was stored. so the writer's index reflects "blocks that were freshly computed" rather than "blocks that are currently in cache." worse, the writer would chain hashes from `parent_block_hash=None` on every fresh batch, producing orphan chains that didn't match the router's canonical chain (which always starts from `parent=0`). those orphan hashes were active noise: lookups would find them, route there, and the engine wouldn't actually have the prefix.

speculative writes turn the dependency around. the router has the tokens in memory at decision time — it doesn't need the engine to tell it what the canonical chain looks like, because the router computed that chain. so after picking a worker, fire-and-forget a write that says "engine X has these blocks." the next request for the same prefix finds them and steers there.

```rust
// after picking a worker, predict that it has (or will soon have) the chain
tokio::spawn(async move {
    speculative_write_chain(&redis_kv, "default", &hashes, &chosen_ip, ttl_secs).await;
});
```

the prediction can be wrong — the engine could have evicted, restarted, or never actually populated the cache. self-correction comes from a 60s ttl: if the prefix isn't seen again within 60s, the entry ages out and the next routing decision falls back to load-only sorting. if the prefix *is* seen again, the lookup itself refreshes the ttl, so hot prefixes self-renew without ever needing a fresh spec write.

## hash chain

both the lookup and the spec write run the same hash function over the same tokens, so they're guaranteed to agree. the chain:

```
block_hash(parent: u64, tokens: &[u32]) -> u64:
    h = blake3()
    h.update(parent.to_le_bytes())     // LE u64, 8 bytes
    for tok in tokens:
        h.update(tok.to_be_bytes())    // BE u32, 4 bytes per token
    return u64::from_le_bytes(h.finalize()[..8])
```

byte ordering is load-bearing. parent is a raw integer computed in memory so we use little-endian; tokens come over the wire so we use network order. the digest is truncated to its first 8 bytes read as little-endian. this lives in one file (`block_hash.rs`, ~30 lines) and both the lookup and the spec-write paths call it. drift is structurally impossible.

trailing partial blocks are dropped — only full `block_size`-sized chunks get hashed. if we hashed the partial, the result wouldn't match what a future request with one more token would produce, so it'd never hit anyway.

## redis structure

```
block:<model>:<rproxy_hash>     SET of engine ips. routing table.
block_access:<model>            ZSET, score=last-access unix ts, member=hash. LRU.
```

that's it. two key kinds.

earlier revisions had `engine:*`, `block_engines:*`, `dedup:*`, `seq:*` to support the zmq writer pipeline (engine→rproxy hash reverse index, per-rank dedupe, replay cursors). once the writer was deleted those keys had no readers — the router only ever consults `block:*` for routing decisions and `block_access:*` for eviction — so they got pulled too.

`block_access` is per-model rather than per-block. one zset per model means eviction is a single `ZRANGEBYSCORE` + `ZREMRANGEBYRANK` pass; a zset per block would explode key cardinality without buying anything.

both spec writes and lookups touch `block_access`. spec writes set the score to "now" because they're predicting a fresh route. lookups bump the score to "now" on any hit, because the prefix is demonstrably still being requested. that bump is what keeps hot prefixes from being evicted.

the kv index lives in its own redis instance, separate from the one holding rate limits and routing config. it churns hard and runs `allkeys-lru`, while the main redis runs `noeviction`. mixing them gets you evicted rate-limit counters at the worst possible moment.

## tokenization

the request side needs `token_ids` to compute the chained hash. easy answer: load a tokenizer in rproxy. wrong answer: chat templates drift. vllm applies its own jinja chat template before tokenizing, and if the router's template diverges by a single whitespace the tokens differ and every block hash misses.

so we don't ship a tokenizer in rproxy. we call vllm's `/tokenize` endpoint on a healthy worker:

```
POST <worker>:8000/tokenize
{
  "model": "<model>",
  "messages": [...],
  "add_generation_prompt": true,
  "chat_template_kwargs": { "reasoning_effort": "low" }
}
```

`add_generation_prompt: true` mirrors what vllm does internally before prefill (including the model's "now generate" sentinel). without it the final block's hash would diverge from the canonical chain by one block on the tail.

`chat_template_kwargs.reasoning_effort` is the gotcha that took the longest to find. the rproxy injects `reasoning_effort=low` into the forwarded chat-completion body, which means vllm tokenizes the actual request with that override active. but the *router's* `/tokenize` call originally sent no kwargs, defaulting to `"medium"`. token 40 in the harmony chat template is the `Reasoning: <effort>` line — token id `4465` for "low" vs `14093` for "medium". block 0 and 1 matched (tokens 0–31 are identical), block 2 onwards diverged, overlap collapsed to wrapper-only. the fix is sending the same kwargs from both paths so they tokenize over the same byte stream.

a subtle property of this: a top-level `reasoning_effort` field on the `/tokenize` request body is silently ignored — it has to live under `chat_template_kwargs`. there's no error, just bad hashes.

## block size

vllm's default block size is 16 tokens; on tpu it's often 64 or 128 because the attention kernel tiles on those boundaries. for our gpt-oss-120b deploy on tpu the value is 16 (confirmed by direct zmq capture of `BlockStored` events, every one had `block_size: 16`). it's currently a `const` in `route.rs`. if we add a model with a different block size, this needs to come from per-model config.

if rproxy's block size disagrees with the worker's by a single token, every chained hash is wrong and overlap collapses to zero. there's no startup probe today — the failure mode is silent, manifested as "kv routing on, hit rate identical to baseline."

## eviction

speculative writes have per-key ttls (60s), but the `block_access` zset only loses members when we explicitly `ZREM`. left alone the zset grows unboundedly even as the underlying `block:*` keys age out. so a periodic sweep runs every 60s per model:

1. **hard cap.** if `ZCARD block_access:<model>` exceeds 500k, evict the lowest-score members until under the cap.
2. **ttl sweep.** evict any member whose score is older than `now - 20min`.

both paths run the same lua script that DELs each victim's `block:*` set and ZREMs the score in one round trip. cap-first ordering matters — doing ttl first could leave us above cap if fewer than `overflow` entries were aged out; cap-first guarantees we land under cap on every call. batches of 500 victims per lua eval keep redis's single-threaded stall bounded under the default 5s script timeout.

## the request pipeline

so the general pipeline whenever we get a request from a user:

1. **tokenize.** post to `<first-candidate-worker>/tokenize` with the request's messages plus `add_generation_prompt: true` and `chat_template_kwargs.reasoning_effort: "low"`. this returns the same tokens vllm will see at prefill.
2. **hash.** chunk the tokens into `block_size`-sized groups, drop the trailing partial, chain-hash each block from `parent=0`.
3. **lookup overlaps.** one lua call against kv redis: for each chain hash, `SISMEMBER block:<model>:<hash> <engine_ip>` for each candidate ip, count hits per ip. on any key with a hit, refresh its ttl and bump its `block_access` score to `now` (this is the "self-renewing hot prefixes" path).
4. **shuffle, then sort** the candidate list. shuffle to break ties randomly across waves; sort by `(is_remote_region, util * 0.5^overlap)` in f64.
5. **spec write.** `tokio::spawn` a write of the canonical chain under the chosen engine's ip with a 60s ttl. fire-and-forget, never blocks the request path, never load-bearing for correctness.
6. **forward** the chat-completion request to the chosen worker.

a few invariants this preserves: the discount is a tiebreaker, not a hard constraint, so no worker ever gets permanently pinned. cold start degrades gracefully to load-only routing because the index is empty and every overlap is zero. and tokenize/lookup failures both fall through to load-only sorting — the kv path is a soft dependency, it must never block a request.
