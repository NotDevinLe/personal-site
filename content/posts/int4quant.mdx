TPUs (Tensor Processing Unit) are from google and are designed for dense linear algebra. Unlike GPUs which are really good at parallel compute in general, TPUs are really really good at linear. This motivates me to write code to support int4 quantized models on them for vLLM. However, TPUs don't natively support int4 datatypes, so we have to write some workarounds.

## how the model is stored

We can store each int4 packed together into a bigger datatype. I chose to store 8 int4s into a int32. To reinterpret this as a int4 model we have the useful function:

``` bash
def ct_u32_unpack_u4(u32_packed: jax.Array) -> jax.Array:
    u4 = jax.lax.bitcast_convert_type(u32_packed, jnp.uint4)
    return jnp.reshape(u4, u4.shape[:-2] + (-1, ))
```

The first line of code says that the computer should treat the int32 model as int4 even though it's really not (and the computer can't actually support it.) Then what happens is that we now have an extra dimension. Why? Because lets say our matrix was something like (n, m), then after expanding it, it becomes (n, m, 8) due to each element being an array. What you do is grab the shape of the matrix except for the last 2 and squash them together by replacing it with -1 which similar to numpy puts all the remaining dimensions into 1. The second line turns it back into (n, m * 8) instead.

Now you might be wondering, "isn't this really inefficient because we're constantly doing bitshifts under the hood?" Well, yes since the TPUs don't support int4, it will do bitshifts to extract each element, but the XLA is like this magic box that will make things a lot faster than if we bitshifted things ourselves.

## implementation

Now we get into the implementation of how do we take a quantized model and dequantize it. We create a class called VllmCompressedTensorsWNA16 which inherits from vLLM's CompressedTensorsScheme class. This class will hold all quantization information such as the zero points, weights, scales, symmetry and other parameters. From this class, there are 5 primary functions.

```bash
create_weights(layer, output_partition_sizes, input_size_per_partition, output_size, input_size, params_dtype, weight_loader)
```

vLLM will call this to instantiate empty layers. Note that this logic is called after the code splits a layer across gpus (if applicable) and the following parameters are per-gpu. `Layer` is litterally the layer we want to modify and add stuff to. `output_partition_sizes` is a list of integers such that [q, k, v] denote the output sizes of each respective matrix. `input_size_per_partition` is the number of input features this gpu is responsible for operating. All of those parameters are for the purpose of megatron-style parallelism which we won't cover in this post.

`PackedvLLMParameter` is a function that allows you to create matrices that pack small datatypes into larger ones (this is exactly what we want for storing our int4s into a int32). We use this to create the weight matrix that work for us. If applicable then we also instantiate empty actorder, scale, and zero point matrices.

```bash
process_weights_after_loading(layer)
```

This function is the one that actually loads everything in and makes things easier for us during the inference time. Firstly, it loads all the matrices that we initialized in the previous section to have their real values. 

After loading, we have to process each weight matrix. Start by unpacking the matrix to be instead of int4 to be int8. The most important thing is that we have to note that columns are out of order in terms of groups. We want all columns to be adjacent to their group members. Therefore, we permute the columns based on actorder. Afterwards, we unpack all other matrices that also need to be unpacked and return this int8 matrix. Notice that we now have an int8 matrix in memory and not an int4. Now, you might be wondering why the heck would we quantize to int4 if we're just gonna dequantize to int8 and have it persist as our model during inference? This is because without doing this, we have no native optimized tpu operations for int4 and significant speed is lost. We also still do gain disk memory improvements, just not while we're serving the model. Finally, save the sorted indices for the columns after we permute them since instead of permuting the weight matrices, we now instead permute the inputs which is much cheaper.

```bash
apply_weights(layer, input)
```

This code is just a router based on whether you fused your Q, K, and V matrices together. If it does, we call `_apply_fused`, otherwise we call `_apply_split`.

```bash
_apply_fused(layer, input, bias)
_apply_split(layer, input, bias)
```

Although they're a bit different, a high level intuition for what they do is the following:
- Permute the input vector. 
- We then do standard dequantization on our persisting weight matrices.

Some notes about this is that although this seems to be just storing int8 in memory and then bloating up each layer during inference to be bfloat16, XLA will detect our operations and make it so that we only expand parts of the matrix up at a time and make it highly ooptimized for speed and memory. This means we aren't ever actually loading in a full bfloat16 matrix.
